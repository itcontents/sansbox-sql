# sqldb-sandbox — Security Notes

Threat model, secret inventory, blast-radius analysis, and rotation
procedures. Read alongside `OPERATIONS.md` (which has the runbook
commands) and `USER_GUIDE.md` (which describes auth from the dev’s
side).

---

## Trust model

The system has three trust boundaries:

1. **Edge / public network** — devs, the ticket-system, the prod bastion.
2. **API host** — runs the FastAPI service under `sandbox` user, owns
   `state.db`, all compose files, TLS material, and the SSH key to the
   bastion.
3. **Production MySQL** — behind the bastion. Reachable only via the SSH
   tunnel opened from the API host using `sandbox_prod_ed25519`.

The API host and prod are different physical/virtual hosts; neither has
direct network trust to the other except via SSH.

## Identity and authentication

| Endpoint group | Required credential | Where the secret lives |
| --- | --- | --- |
| `POST /instance`, `GET/PATCH/DELETE /session/...`, `GET /session-tls/.../ca.pem` | `X-API-Key: <SANDBOX_API_KEY>` | `.env` (mode 0640, owner `root:sandbox`); rotated by ops |
| Same endpoints, when Cloudflare Access is configured | `Cf-Access-Jwt-Assertion: <jwt>` (signed by CF) | not stored locally; verified via `.well-known/jwks.json` from CF |
| `POST /webhook/ticket-closed` | HMAC `X-Webhook-Signature: sha256=<hex hmac of body with SANDBOX_WEBHOOK_SECRET>` | `.env` |
| `mysqldump` against prod | `PROD_MYSQL_USER=sandbox_dump` + `PROD_MYSQL_PASSWORD` via `MYSQL_PWD` env var | `.env` |
| SSH tunnel to bastion | `~/.ssh/sandbox_prod_ed25519` (Ed25519, mode 0600) | created and rotated by ops |
| Sandbox MySQL access (per-DB) | per-session `user` + `password` returned to dev in the create response | encrypted at rest in `state.db` via Fernet |

Notes:

- `--ssl-mode` on the **prod** side is `DISABLED` because SSH is the
  transport. Cipher selection is SSH’s responsibility.
- The sandbox MySQL forces TLS (`require-secure-transport = ON`) and
  refuses plain-text connections.
- Every API host request is logged by `journalctl -u sqldb-sandbox`,
  including failed-auth attempts and category/error from any failed
  create. There is no separate audit DB.

## Secret inventory and rotation cadence

| Secret | Where | Generated by | Lifetime | Rotation |
| --- | --- | --- | --- | --- |
| `SANDBOX_API_KEY` | `.env` | `bootstrap.sh` (`openssl rand -hex 32`) if placeholder | indefinite, but treat as 90 days | two-step: `.env`, then consumers; restart service |
| `SANDBOX_WEBHOOK_SECRET` | `.env` | `bootstrap.sh` (`openssl rand -hex 32`) | indefinite | rotate on ticket-system onboarding changes |
| `SANDBOX_FERNET_KEY` | `.env` | `bootstrap.sh` (`Fernet.generate_key()`) | long-lived, but rotation is destructive | snapshot state.db → nuke active sessions → rotate → restart |
| `SANDBOX_NGINX_SSL_CERT` / `_KEY` | external path | Let’s Encrypt or CF Origin CA | 90 days (LE) or 10y (CF Origin) | `certbot renew` for LE; rolling reissue for CF |
| `PROD_MYSQL_PASSWORD` | `.env` (and any Vault equivalent) | operator at prod | per prod policy | rotate at prod; pass to API via `.env` and restart |
| Prod SSH private key | `/home/sandbox/.ssh/sandbox_prod_ed25519` (mode 0600) | operator | per prod policy | generate new key, add `.pub` to bastion, swap file atomically, restart |
| Per-session `ca.key` | `/var/lib/sandboxes/tls/<sid>/` | created on `POST /instance`, never leaves the API host | container lifetime | nuked when session expires |

Rotate the per-session TLS CA encryption key only if you rotate
`SANDBOX_FERNET_KEY` — fernet-encrypted credentials live in `state.db`,
not the CA bundle. If the Fernet key changes, neither old sessions’
credentials nor new ones sharing keys will decrypt.

## Blast radius — what each leaked credential unlocks

### `SANDBOX_API_KEY` (32-byte hex)

Attacker can:
- create, view, reset-ttl, nuke, and download CA certs for **any**
  sandbox they know the `sid` of (sids leak in `state.db` admin dumps
  or in the dev’s clipboard).
- read `ca.pem` for any active session, then connect to that session
  from anywhere on the public network with the per-DB creds they pull
  from `state.db` if they also have read access there.
- cannot read existing `state.db` directly (it’s a sqlite file the API
  reads, not the dev), but can read session details via `GET
  /session/{sid}`.

Attacker cannot:
- reach the prod bastion or prod MySQL (no prod creds in this leak).
- decrypt per-DB credentials already in `state.db` (they need
  `SANDBOX_FERNET_KEY`).

### `SANDBOX_WEBHOOK_SECRET`

Attacker can issue fake `ticket-closed` webhooks that **nuke any
session** (`POST /webhook/ticket-closed`). They cannot create sessions
and cannot read state.

Mitigation: if a webhook integration is misbehaving, rotate the secret;
legitimate webhook callers update on their side.

### `SANDBOX_FERNET_KEY`

Attacker can:
- read the per-DB passwords out of `state.db` and connect to live
  sandboxes by their published `mysql_host`/`mysql_port`.
- past sandboxes that are nuked but whose session rows are not yet GC’d
  from `state.db` are still decryptable.

Attacker cannot:
- read prod credentials (those are not encrypted with Fernet).
- start new prod tunnels (prod SSH key is separate).

Rotation is destructive: re-encrypt every row through the old key, then
swap. Simpler: nuke all active sessions, rotate, then devs recreate.

### `PROD_MYSQL_PASSWORD` + prod SSH key (both required)

Attacker can: connect to prod MySQL via the bastion as the dump-only
`sandbox_dump` user, then perform `mysqldump`. They cannot connect
directly to prod from outside the bastion because prod MySQL
(`192.168.0.50:3306`) is on a private network only reachable from
the bastion.

Both must leak for prod impact. Rotate at prod:
1. Revoke `sandbox_dump` on prod, issue new password.
2. `PRIVILEGES` for `sandbox_dump` should already be locked to `SELECT`
   on the required DBs only (verify at deploy: `SHOW GRANTS FOR
   'sandbox_dump'@'%';`).

### Per-session CA private key (in `/var/lib/sandboxes/tls/<sid>/ca.key`)

Attacker can: MITM an active session if they can also intercept
network traffic. Per-session, time-boxed impact. The CA key is never
exposed over the wire — only the public `ca.pem` is served. Compromise
requires on-API-host file read.

Mitigation: keep `mode 0600` on the `tls/` dir; the per-session key is
ephemeral and dies with the session.

### Prod SSH key

Attacker can: log into the bastion as the `sandbox` user. From there
they inherit the bastion’s network reach — typically the prod MySQL
private network.

Mitigation: rotate at bastion, `ssh-keygen -t ed25519` produces a new
key. Roll the old public key out of `authorized_keys`.

### Total compromise of the API host (root)

Attacker has all of the above, plus:
- read all `state.db` rows (decryptable with fernet key from `.env`).
- read all per-session TLS material (public + private keys).
- impersonate the API service indefinitely.

Mitigation: host hardening is the same as any other sandboxed service:
private network for SSH, OS patching, restricted operator accounts,
disk-encryption at rest. Boot integrity beyond that is out of scope.

## Encrypted at rest

| Data | Encryption | Where | Lifetime |
| --- | --- | --- | --- |
| Per-DB password columns in `state.db` | Fernet (`SANDBOX_FERNET_KEY`) | `/var/lib/sandboxes/state.db` | nuke session → creds still in row until reaper vacates; see `store.claim_expired` |
| Per-session TLS material | plaintext on disk | `/var/lib/sandboxes/tls/<sid>/` | container lifetime |
| `state.db` itself | none (sqlite file; relies on host fs encryption) | `/var/lib/sandboxes/state.db` | forever |
| MySQL dumps during create | never to disk | `pipe` between `mysqldump` and `mysql` | process lifetime |
| Per-session log files | none | `/var/log/sandboxes/<ticket>-<date>/<sid>/` | forever (logrotated) |
| `.env` | none on disk (server-side secrets) | `/opt/sqldb-sandbox-setup/.env` (mode 0640, owner `root:sandbox`) | forever |

The **dump never lands on disk** — `mysql_ops.clone_db` pipes
`mysqldump` straight into `mysql` on the sandbox. There is no
intermediate `.sql` file under `/tmp` or `/var/lib`. Files under
`/var/lib/sandboxes/dump-*.sql` from older versions were a
workaround that’s been removed.

## Per-session isolation guarantees

Each sandbox container has:
- `cap_drop: [ALL]` + `security_opt: [no-new-privileges]` — no
  capabilities beyond defaults, no privilege escalation.
- `pids_limit: 256` — runaway processes can’t spawn.
- `mem_limit: 2g` — OOM-killed before it can starve the host.
- `cpus: 1.0` — host scheduler never starves sibling tenants.
- random root password (`gen_root_password()` in `docker_ops`).
- per-session TLS cert and CA, no global keys.
- per-DB scoped user (`u_<db>_<8hex>`) with grants limited to that
  one DB; not a super-user.

Sandbox runs `mariadbd --user=999:999` (the `mysql` user inside the
container). The host maps that to a non-root UID on the host. Bind-mount
of `/var/log/mysql` is read/write but isolated from the host’s MySQL.

The API host’s docker socket is `0660 root:docker`. The `sandbox` user
is in `docker`; `lytadmin` is also in `docker`. Whoever can run docker
on the host can effectively become root inside any container — that’s
how `sg docker` works. Treat docker-socket access as root.

## API process sandbox

The systemd unit hardens:

- `NoNewPrivileges=true` — no `setuid` magic for the API process.
- `PrivateTmp=true` — `/tmp` is private.
- `ProtectSystem=full` — most FSes are read-only.
- `ReadWritePaths=/var/lib/sandboxes /var/log/sandboxes` — only those
  two paths are writable.
- `MemoryMax=2G` / `CPUQuota=200%` / `TasksMax=512` — runaway
  `mysqldump` or restore can’t take the host down.
- `User=sandbox` / `Group=sandbox` — runs as the unprivileged service
  user, not root.

The API host never runs the service as root in steady state.

## Network posture

- The API listens on `127.0.0.1:8080`. External reach comes from either
  `nginx` (with `deploy/nginx/sqldb-sandbox.conf` rendering as
  `/etc/nginx/sites-available/sqldb-sandbox.conf`) or Cloudflare
  Tunnel.
- Cloudflare Access is the optional `Cf-Access-Jwt-Assertion` gate;
  when configured, **all** endpoints except the webhook require it.
- The webhook endpoint intentionally skips CF Access (because the
  ticket-system is a different upstream), but signs its body with
  HMAC.
- Developers reach the API at `https://${SANDBOX_PUBLIC_HOST}/…`,
  terminating TLS at nginx or CF.
- `--proxy-headers` is enabled in uvicorn and the systemd `ExecStart`
  passes `--forwarded-allow-ips='*'` so nginx origin IP appears in
  `journalctl`. The `forwarded-allow-ips` is open because we’re
  fronted by nginx on a private loopback here; tighten it to
  `127.0.0.1` in deployments where uvicorn is internet-facing.

## What the system is **not** designed to do

These are explicit non-features:

- **No cross-DB joins or cross-session sharing.** Each session is
  one container; per-DB grants are scoped; no “join across tickets”
  workflow exists.
- **No migration back into prod.** The sandbox is a debugging tool.
  There is no write-back path from sandbox to prod by design.
- **No audit-grade access logs.** `journalctl` is fine-grained enough
  for individual incidents but is not designed for compliance-grade
  long-term storage. If you need it, forward `journalctl -u
  sqldb-sandbox` into a SIEM and rotate it off-host.
- **No defense against a compromised docker socket.** Whoever can
  run docker on the host effectively has root. Consider:
  - dedicated docker-socket group that excludes `lytadmin`/the
    user that runs bootstrap,
  - `--userns-remap` if you want to remap uids out of root.

## Incident checklist

If you suspect the API host or any secret is compromised:

1. **Snapshot.** `systemctl stop sqldb-sandbox && sqlite3 .backup
   /var/lib/sandboxes/state.db /tmp/state-snapshot.db` and keep the
   journal for forensics.
2. **Rotate all four env-managed secrets** at once: `SANDBOX_API_KEY`,
   `SANDBOX_WEBHOOK_SECRET`, `SANDBOX_FERNET_KEY`,
   `PROD_MYSQL_PASSWORD`. (Rotating one in isolation is rarely worth
   the cost of chasing who-else.)
3. **Re-issue the prod SSH key.** Revoke the old `ed25519` from
   `bastion:/home/sandbox/.ssh/authorized_keys`.
4. **Nuke every active session** with `curl -X DELETE
   /session/<sid>` for each, or nuke `state.db` rows and let the
   reaper clean up containers. Even dev connections won’t be able to
   decrypt old sessions after a Fernet-key rotation.
5. **Rebuild.** `docker compose down -v`, delete the per-session
   dirs under `/var/lib/sandboxes/{composes,tls,log}/`, and verify
   `state.db` is empty.
6. **Re-bootstrap from a clean state** by re-running `bootstrap.sh`
   with fresh secrets.
7. **Postmortem.** Review `journalctl` for the time window from the
   suspected intrusion to incident-response start. Capture the
   attacker’s IPs from the audit log lines, not from network logs.
